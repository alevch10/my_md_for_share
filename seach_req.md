# Единый поиск по сайту — функциональные требования

Версия 1.0 · Платформа: 1С-Битрикс: Управление сайтом · Поисковый движок: Elasticsearch

---

## 1. Цель

Единый поисковый модуль, работающий из любой точки сайта. В каждом разделе (Цены, Услуги, Врачи, Блог, Роддом, Клиники) поиск выдаёт целевые результаты этого раздела в приоритете и дополнительно показывает релевантные альтернативы из других разделов.

**Целевая метрика:** конверсия из поиска — доля переходов из результатов поиска на карточку услуги / врача / акции с последующим целевым действием (запись, звонок, заявка). Метрика отслеживается через события Яндекс.Метрики (см. раздел 8).

## 3. Архитектура и технологии

- **Платформа:** 1С-Битрикс: Управление сайтом. Все данные (услуги, врачи, блог, акции, отделения) хранятся в единой БД, в разных инфоблоках с разной структурой полей.
- **Поисковый движок:** Elasticsearch (действующая инсталляция).
- Индекс строится по полям заголовка и основного текстового содержимого документа; служебные поля (alt/title изображений, meta-теги, технические атрибуты) в индексацию не включаются.
- Весовые коэффициенты полей: совпадение в заголовке весит существенно больше совпадения в теле текста (ориентир: заголовок ×5–10 относительно тела).
- Обязательное поле индекса `doc_type` (тип документа) с весовым коэффициентом на уровне типа — см. таблицу 4.2.
- Синонимы реализуются через synonym token filter, опечатки — через fuzziness (штатные средства Elasticsearch).

## 4. Логика ранжирования

### 4.1. Формула

```
score = (text_relevance × type_weight) + boost_current_section + boost_exact_title_match
```

#### 4.1.1 Расшифровка элементов 

| Параметр | Перевод | Описание |
|---|---|---|
| text_relevance | текстовая релевантность | Это стандартная оценка Elasticsearch: насколько текст запроса совпадает с текстом документа (заголовок, содержимое, синонимы). Чем точнее и чем более редкие/значимые слова совпали — тем выше число. Это единственный компонент, который считает не человек, а поисковый движок автоматически по своим алгоритмам (BM25). |
| type_weight | вес типа раздела | Ручной коэффициент важности, который мы задали (См. 4.2.). Он умножается на текстовую релевантность — то есть даже при одинаковом текстовом совпадении статья блога всегда получит меньше баллов, чем услуга, просто потому что у типа «Блог» низкий множитель. |
| boost_current_section | бонус за текущий раздел | Фиксированная добавка, которая начисляется, только если документ относится к тому разделу сайта, откуда человек начал поиск. Гарантирует, что раздел пользователя окажется в топе, даже если по чистой текстовой релевантности что-то другое было бы выше. |
| boost_exact_title_match | бонус за точное совпадение заголовка | Ещё одна фиксированная добавка — начисляется, если запрос почти дословно совпал с заголовком документа (например, запрос «МРТ Голени» и услуга называется «МРТ Голени»). Это защита от ситуации, когда короткое слово упомянуто много раз в длинной статье блога (за счёт этого у неё высокая text_relevance), но в реальности человек искал не статью, а конкретную услугу с таким названием |

### 4.2. Веса по типу документа (вне контекста текущего раздела)

| Тип документа | Вес | Комментарий |
|---|---|---|
| Услуги | 10 | Максимальный приоритет по умолчанию |
| Врачи | 9 | |
| Акции | 8 | |
| Отделения / Клиники | 7 | |
| Сервисные страницы | 7 | Личный кабинет, калькулятор, ДМС и т.п. — см. 5.3 |
| Памятки / Лендинги | 6 | Инструкции/памятки к процедурам — см. 6.2. Вес выше блога, так как это точный целевой ответ на конкретный запрос, а не общая статья |
| Блог / Новости | 3 | Низкий приоритет вне раздела «Блог» |

### 4.3. Приоритет текущего раздела (scoped-поиск)

Если поиск выполняется внутри конкретного раздела, результаты этого раздела получают буст и показываются первым блоком, независимо от базового веса типа.

- Поиск в разделе «Блог» → в топе статьи блога, ниже — альтернативы (услуги, врачи по теме).
- Поиск в разделе «Услуги» или на главной странице → в топе услуги, ниже — альтернативы (врачи, статьи).

### 4.4. Буст точного совпадения заголовка

Короткие однословные/двусловные запросы (МРТ, УЗИ, Гинеколог) должны в первую очередь вести на карточки услуг/специализаций с точным совпадением в заголовке, а не в статьи блога, где термин упомянут многократно в тексте. Точное или почти точное совпадение запроса с заголовком документа получает существенный дополнительный буст сверх базовой текстовой релевантности.

### 4.5. Ранжирование внутри блока «Врачи»

Сортировка результатов внутри блока «Врачи» переиспользует логику, уже применяемую на странице раздела «Врачи» (стандартная фильтрация/сортировка каталога). Отдельная логика ранжирования врачей для поиска не разрабатывается.

### 4.6. Приоритет блока при точном совпадении в разделе с меньшим базовым весом

На странице результатов поиска, вне текущего раздела (см. 4.3), порядок блоков по умолчанию соответствует весам из таблицы 4.2. Если в разделе с более низким базовым весом найдено точное (или почти точное) совпадение заголовка сущности с запросом (см. правило буста в 4.4), а в разделах с более высоким базовым весом — только частичное совпадение, блок с точным совпадением поднимается выше блоков с более высоким базовым весом, но без точного совпадения.

Пример: по запросу «Подготовка к колоноскопии» блок «Памятки/Лендинги» (базовый вес 6) содержит точное совпадение заголовка, а блок «Услуги» (базовый вес 10) — только частичное («Колоноскопия» без слова «подготовка»). В этом случае блок «Памятки/Лендинги» показывается выше блока «Услуги».

Правило не действует на раздел, из которого выполнялся поиск, — он всегда занимает первую позицию (см. 4.3, раздел 10.3) независимо от наличия точных совпадений в других разделах.

## 5. Специальные сценарии

### 5.1. Поиск по ФИО врача

Поиск должен уверенно распознавать форматы запроса:

- Только фамилия (Величко)
- Полное ФИО (Величко Ирина Сергеевна)
- Формат «Фамилия И.О.» (Величко И.С.)
- Специализация + фамилия (Невролог Величко)

При совпадении по фамилии/ФИО всегда показывается страница результатов поиска (без прямого редиректа на карточку врача) — это оставляет пользователю доступ к форме обратного звонка и альтернативам на случай ошибочного совпадения.

Если запросу соответствует несколько однофамильцев — в выдаче показываются все, сортировка стандартная (см. 4.5).

### 5.2. Нетематические / сервисные запросы

Запросы, не относящиеся к услугам, врачам или контенту, а ведущие на функциональные страницы сайта («Личный кабинет», «Мобильное приложение», «Налоговый вычет», «ДМС», «ОМС» и т.п.), обрабатываются отдельным типом документа «Сервис/Страница» с весом, аналогичным Отделениям (см. 4.2), чтобы такие запросы уверенно попадали в топ.

### 5.3. Взрослые / дети

Явное разделение выдачи «для взрослых» / «для детей» и явные фильтры/фасеты (взрослый-ребёнок, специализация и т.п.) в выдаче результатов поиска не реализуются — это сузило бы предложение на этапе поиска. Уточнение происходит на целевой странице раздела.

Определение контекста «ищу для ребёнка» по тексту запроса (например, «педиатр», «детский массаж») в MVP не реализуется — вопрос вынесен в раздел 12 как открытый на будущее.

## 6. Синонимы и терминология

### 6.1. Принцип: синонимы задаются на уровне сущности, а не в общей таблице

Синоним всегда привязан к конкретной сущности конкретного типа (услуга, памятка/лендинг и т.п.), а не хранится как абстрактная пара «термин → термин» в отдельной центральной таблице. У каждой индексируемой сущности в её карточке в административном интерфейсе есть собственное поле «Синонимы/варианты запроса» (алиасы), которое заполняет ответственный за эту сущность контент-редактор — например, редактор каталога услуг сразу видит и заполняет синонимы прямо на странице редактирования услуги.

При поиске значения из поля синонимов индексируются вместе с сущностью (наравне с заголовком, с высоким весом, аналогичным весу заголовка — см. 3), поэтому запрос находит нужную сущность через штатный полнотекстовый поиск Elasticsearch, без отдельного шага «расширения запроса» и без специальной логики, привязывающей один тип запроса к нескольким типам сущностей одновременно.

Примеры:

| Запрос пользователя | Сущность, к которой привязан синоним | Тип сущности |
|---|---|---|
| Колоноскопия | Видеоколоноскопия диагностическая | Услуга |
| Флюорография | Рентгенография грудной клетки (1 проекция) | Услуга |
| Окулист | Первичный приём офтальмолога | Услуга |
| Лор | Первичный приём отоларинголога | Услуга |
| Подготовка к колоноскопии | Подготовка к эндоскопическим исследованиям и ирригоскопии | Памятка/лендинг |

Если по такому запросу в выдаче помимо услуги дополнительно естественным образом оказываются и врачи нужной специальности (например, окулист → показана услуга «Первичный приём офтальмолога», ниже — врачи-офтальмологи) — это происходит без дополнительной логики в поиске: специальность врача уже упоминается в его карточке/теге как обычный текст, поэтому попадает в выдачу через стандартное текстовое совпадение, если синоним привязан и к нужному термину в целом (см. 6.2 про общий словарь). Отдельная функция «специальность → услуга + врачи» в поиске не разрабатывается.

### 6.2. Памятки/лендинги как отдельный тип сущности

Инструкции и памятки к процедурам (например, «Подготовка к эндоскопическим исследованиям и ирригоскопии») выделяются в отдельный тип документа «Памятка/лендинг» — не блог и не услуга (см. таблицу типов в разделе 2, вес — в 4.2). У сущностей этого типа так же, как у услуг, есть собственное поле синонимов, заполняемое редактором.

### 6.3. Общий словарь — для формулировок, не привязанных к одной сущности

Часть запросов не сводится к одной конкретной сущности: сокращения, жаргон и опечатки, которые должны нормализовываться независимо от того, к какой именно услуге/специальности они в итоге приведут (Флг → флюорография, Кт → КТ, Чек-ап → чекап, НИПТ, ВПЧ, КТГ). Для них сохраняется общий словарь нормализации на уровне поискового движка (synonym token filter Elasticsearch) — он не заменяет сущностные синонимы из 6.1, а дополняет их для случаев, когда конкретная сущность-получатель не определена заранее.

Разговорные формулировки и симптомы (см. 5.2) также ведутся через этот общий словарь, поскольку один симптом обычно ведёт не к одной сущности, а к комбинации из нескольких (врач + услуга).

### 6.4. Механизм пополнения

1. Сущностные синонимы (6.1) вносятся редактором конкретного раздела прямо в карточке услуги/памятки/другой сущности — отдельного централизованного интерфейса для них не требуется.
2. Общий словарь (6.3) ведётся через административный интерфейс (кастомный инфоблок / HL-блок Битрикса) отдельным ответственным сотрудником.
3. Источник пополнения обоих слоёв — логи поисковых запросов (раздел 8): в первую очередь запросы с нулевой выдачей и запросы с низким CTR по топ-5 результатам.
4. Стартовое наполнение — на основе собранного топ-100 запросов (приложение к документу).

## 7. Опечатки и транслит

- Поиск должен прощать опечатки и находить ближайшие по написанию слова (fuzzy search) — реализуется штатными средствами Elasticsearch (fuzziness).
- Поддержка транслита и неверной раскладки клавиатуры (например, «jrekbcn» → «окулист») желательна, но не обязательна для MVP — реализуется, если не увеличивает существенно сроки/сложность.

## 8. Логирование и аналитика

### 8.1. Что логируется

- Каждый поисковый запрос (текст, раздел, из которого выполнен поиск, дата/время).
- Количество результатов в выдаче — по каждому типу документа отдельно и в топ-блоке в целом (включая фиксацию запросов с нулевой выдачей).
- Выбор пользователя из выдачи: по какому результату кликнули, из какого блока (топ-раздел/альтернативы), на какой тип документа.
- Целевое действие после перехода (запись на приём, звонок, заявка) — там, где технически возможно отследить.

### 8.2. Способ реализации

Приоритетный вариант — события Яндекс.Метрики (goal / eventLabel на каждый этап: поиск выполнен → показана выдача → клик по результату → целевое действие). Отчёты и графики строятся штатными средствами Метрики, без разработки отдельной аналитической панели. Логирование на бэкенде — опциональный дублирующий вариант при необходимости более глубокой аналитики.

## 9. UX и представление результатов

Общие принципы; детальная спецификация страницы результатов, подсказок и их поведения — в разделе 10.

- Поисковая строка и логика поиска — единый компонент на всех страницах сайта (см. 10.1).
- **Топ-5 на блок:** по каждому релевантному разделу показывается не более 5 результатов с кнопкой «Все», переводящей в развёрнутую выдачу этого раздела (см. 10.3–10.4).
- **Порядок блоков:** приоритет — разделу, из которого выполнен поиск; вне этого раздела порядок определяется весами из таблицы 4.2 с поправкой на точные совпадения (см. 4.6 и 10.3).
- **CTA в карточках результатов:** в карточках услуг и врачей в выдаче поиска предусмотрена кнопка целевого действия («Записаться» и т.п.) без обязательного перехода на страницу карточки.
- **Автокомплит/подсказки:** реализуются при вводе запроса — см. полную спецификацию в 10.2.
- **Нулевая выдача:** при отсутствии результатов (в т.ч. после нечёткого поиска) показывается сообщение «Ничего не найдено» и форма обратного звонка (см. 10.5).
- **Фильтры/фасеты:** не предусмотрены на смешанной странице результатов по всем разделам; доступны при переходе в развёрнутую выдачу конкретного раздела (см. 10.4).

## 10. Страница результатов поиска и подсказки

### 10.1. Поисковая строка

- Поисковая строка присутствует на всех страницах сайта, в верхней части.
- На мобильных устройствах при переходе в фокус поля открывается стандартная системная клавиатура ввода.

### 10.2. Подсказки при вводе (автокомплит)

- Подсказки формируются в реальном времени по мере ввода запроса, с задержкой 500 мс от последнего нажатия клавиши — чтобы не отправлять запрос на бэкенд на каждый символ.
- Состав подсказок: направления, услуги, специальности, ФИО врачей, заголовки памяток/лендингов, акции. Сервисные страницы (личный кабинет, калькулятор и т.п.) в подсказки не включаются — см. открытый вопрос 12.
- Всего подсказок — до 10. Видимое без прокрутки количество ограничено высотой в две строки (варьируется в зависимости от длины текста подсказок); остальные скрыты под кнопкой «Ещё».
- Сортировка подсказок — по степени текстового совпадения запроса с найденным вариантом: чем ближе совпадение, тем выше позиция в списке. Точная технико-числовая реализация («процент совпадения символов») зависит от возможностей скоринга Elasticsearch (fuzzy score / edit distance) — см. открытый вопрос 12.
- Клик по подсказке выполняет полноценный поиск по тексту подсказки — пользователь попадает на страницу результатов поиска (10.3), а не сразу на карточку конкретной сущности. Это осознанное решение: клик по специальности может означать желание увидеть либо весь список врачей направления, либо конкретно услугу первичного приёма, — прямой переход на одну карточку лишил бы пользователя выбора (та же логика, что и в сценарии поиска по ФИО врача, см. 5.1).
- Кнопка «Найти» выполняет тот же поиск, но по тексту, введённому пользователем вручную (без выбора подсказки).

### 10.3. Передача контекста и общая страница результатов

- При выполнении поиска (клик «Найти», клик по подсказке, Enter) в запрос передаётся раздел сайта, из которого был инициирован поиск.
- Страница результатов показывает блоки по разделам: Услуги, Врачи, Акции, Отделения/Клиники, Памятки/Лендинги, Сервисные страницы, Блог. Состав соответствует структуре типов документов сайта (см. раздел 2).
- Для каждого блока отображается заголовок раздела и количество найденных результатов.
- Раздел, в котором результатов не найдено, на странице выдачи не показывается.
- Порядок блоков по умолчанию — по весам из таблицы 4.2 (Услуги → Врачи → Акции → Отделения/Клиники → Памятки/Лендинги → Сервисные страницы → Блог), с поправкой на точные совпадения — см. правило 4.6.
- Раздел, из которого выполнялся поиск, всегда занимает первую позицию среди блоков — правило имеет более высокий приоритет, чем базовые веса и правило точного совпадения (см. 4.3, 4.6).
- Внутри каждого блока показывается не более 5 результатов и кнопка «Все», переводящая в развёрнутую выдачу этого раздела (см. 10.4).
- Карточки результатов внутри блока визуально идентичны карточкам на стандартной странице соответствующего раздела: карточка врача в выдаче поиска выглядит так же, как карточка врача на странице «Врачи» — и аналогично для остальных типов (Услуги, Акции, Отделения, Блог).
- Фильтры/фасеты на этой смешанной странице (по всем разделам сразу) не предусмотрены.

### 10.4. Развёрнутая выдача раздела

- По клику «Все» пользователь переходит в развёрнутую выдачу выбранного раздела — тот же результат поиска, но только по одному разделу, с постраничной пагинацией.
- В развёрнутой выдаче раздела доступна фильтрация, аналогичная фильтрации на обычной странице этого раздела (например, для «Врачей» — фильтр по специальности и клинике). Технически это переиспользование той же логики фильтрации и пагинации, что уже есть на стандартной странице раздела, только предзаполненной результатами поиска, а не полным каталогом.
- Фильтры и пагинация присутствуют только на этом уровне (развёрнутая выдача одного раздела) — не на смешанной странице результатов (10.3), что соответствует правилу из 9.

### 10.5. Форма «Не нашли что искали»

- Единая форма заказа обратного звонка «Не нашли что искали?» показывается под результатами / под пагинацией.
- Используется один и тот же компонент формы как при частичной выдаче (когда что-то найдено, но не то, что нужно), так и при полностью нулевой выдаче (см. 9) — отдельная форма для случая нулевой выдачи не создаётся.

## 11. Вне рамок текущего проекта

- Полноценный интерактивный симптом-чекер (дерево вопросов, отдельный UI-сценарий) — только словарные связки симптом → врач/услуга в MVP (см. 5.2).
- Гео-выдача и мультиклиничная фильтрация.
- Явные фасетные фильтры на смешанной странице результатов поиска (взрослый/ребёнок и т.п.) — фильтрация внутри развёрнутой выдачи одного раздела в объём входит, см. 10.4.
- Автоматическое построение словаря синонимов через ML/эмбеддинги без участия редактора.

### 11.1. Поиск по симптомам (MVP)

Реализуется как расширение общего словаря синонимов/связок (раздел 6.3), без отдельного диалогового «симптом-чекера».

- Симптомный запрос может вести к нескольким результатам одновременно: специализация(и) врача + релевантная(ые) услуга(и)/анализ(ы).
- Пример: «болит живот справа внизу» → Гастроэнтеролог, Хирург, УЗИ брюшной полости.
- Наполнение словаря — по частотным симптомным запросам из логов поиска; приоритет — частотным формулировкам, а не попытке покрыть все возможные симптомы сразу.
- Полноценный интерактивный симптом-чекер (дерево вопросов, отдельный UI-сценарий) в объём текущего проекта не входит — см. раздел 11.
